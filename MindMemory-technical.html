MindMemory
Technical brief · for engineers and researchers

Memory for Claude Code.

A drop-in folder + Python hooks that give Claude Code a regenerable, markdown-native memory layer with continuous self-maintenance and a built-in measurement harness.

CoMindLab Labs · Research preview · MIT · April 2026

01What it is

MindMemory is a portable memory layer for Claude Code. Three primitives:

  • Markdown files in a structured layout — no vector DB, no service.
  • Five Python hooks wired to Claude Code's PostToolUse, SessionStart, and SessionEnd events.
  • One setup wizard skill that bootstraps a project with the right structure in ~5 minutes.

Total dependencies: zero. Standard library only. Read the source — it's the entire system.

Repository: github.com/CoMindLab-ai/mind-memory

03The four tiers

Memory is organised by lifespan, not by topic. This maps cleanly onto "what gets loaded at session start" vs "what gets retrieved on demand" vs "what stays out of context entirely."

TierFileLifespanRole
1 · Identityidentity/anchor.mdRead-onlyPersonality. Hard rules. Loaded every session.
2 · Long-termmemory/MEMORY.md + {topic}.mdWeeks+Confirmed patterns. Indexed — index always loaded, details on demand.
3 · Workingmemory/working-memory.mdDaysThis week's focus. Token-budgeted (~5KB). Rolls into Tier 2 or 4.
4 · Archivememory/archive.mdRarely readCompressed historical context. Searchable, not loaded by default.

The promotion rule. Borrowed from scientific reasoning to keep memory curated:

  1. 1× — observation, logged but not in memory
  2. 2× — hunch (tentative, labelled in working memory)
  3. 3× — heuristic (promoted to Tier 2, applied)
  4. Contradicted — demoted to hunch or archived with note

Each Tier-2 entry stores both why (the reasoning behind the rule) and how to apply (concrete trigger). The "why" lets the model judge edge cases instead of blindly applying.

04The three loops — continuous self-maintenance

Most "AI memory" systems consolidate at session end. MindMemory's defining design choice: memory curates itself mid-session, not just at logoff.

INNER LOOP   every ~10–25 tool calls (PostToolUse hook)
  memory-collector → scan transcript, capture signals
  memory-curator   → promote / demote per 3× rule

MIDDLE LOOP  per session
  SessionStart → load identity + MEMORY.md + working-memory
  SessionEnd   → write metrics row to CSV

OUTER LOOP   weekly (7-day timestamp gate)
  metrics-brain      → compute H1–H5, write daily-report.md
  memory-index       → regenerate INDEX.md (optional feature)

Why nest them? By the time a session ends, the model has drifted from the moment of insight. The inner loop catches corrections within minutes, producing tighter, more specific rules than catching them hours later. Bookends and weekly review are safety nets and measurement.

05The signals collector + curator detect

SignalPatternWhat it triggers
Correction"no", "wrong", "don't do that"Candidate new heuristic
Confirmation"yes exactly", "perfect", "keep doing that"Reinforce existing approach
Follow-throughUser accepts output without changesSilent confirmation
Frustration"again?", "as I said", strong negative languageEscalate severity in working memory
Repeated topicSame correction 2+ times in a sessionAuto-promote to hunch

Pattern-matching only — no LLM call inside the collector. The curator does call claude -p for promotion judgment when a hunch hits 3× confirmation, because that's where world-knowledge actually helps.

06Measurement — the H1–H5 hypotheses

Every install ships with a measurement harness. Five hypotheses, computed from the session-metrics CSV:

  • H1 — Production ratio: productive_calls / total_calls (↑ if memory helps)
  • H2 — Orientation: exploration calls before first edit (↓ if memory helps)
  • H3 — Agent efficiency: agent delegations / total calls (↓ if memory helps)
  • H4 — Density: productive calls per session (↑ if memory helps)
  • H5 — Repeat-correction rate: % sessions with repeated corrections (↓ if memory helps) — primary outcome

Run python .claude/hooks/metrics-brain.py after two weeks of use to see your own H1–H5. The system measures itself; users self-test whether memory is actually helping their work.

What's robust: H1, H2, H4 are large effects, robust to a sensitivity check excluding the W15 merge window (where a parallel-implementation merge inflated repeat-corrections regardless of memory state).

What's null: H5 — the headline outcome — has not been demonstrated. Three honest theories in the findings doc:

  • Selection bias — memory loads on harder sessions; the control group is biased toward easy work
  • Coverage gap — memory doesn't have the right lessons yet
  • Recall gap — memory exists but isn't applied under pressure

Cannot separate them with current n. Publishing the result is itself the contribution: the system that measures itself is the only honest way to test "does memory in AI work?"

The interesting bit for reviewers: the qualitative report from the n=1 user ("you remembering things helps me") is consistent with H1/H2/H4 but invisible to H5. Format/pacing preferences (dyslexia, visual-learner) get used silently every session — no "wrong"/"again" language fires. The most valuable memory is the kind H5 can't detect.

08Install — three commands

cp -r starter-kit/. your-project/
cd your-project
mv .claude/settings.json.example .claude/settings.json
claude
> /mind-memory-setup

The wizard asks 7 questions (with deep-dive opt-in for richer personality), respects existing files via an archive mechanism, never destructively modifies CLAUDE.md (uses a managed marker block), and writes to project-only / global / auto-split scope per your choice.

09What we want from reviewers

  1. Critique the experimental design. If H5 is the wrong primary outcome, tell us what's better. 04-open-questions.md has the full list.
  2. Run it on your work, share the CSV. One user's data is anecdotal. Twenty users' data starts to be evidence.
  3. Find counter-examples. "I put this in memory and Claude still didn't apply it" is gold. Specific cases beat general theories.

10Lineage and license

The kernel is Andrej Karpathy's LLM Wiki framing — an agent that writes down what worked and what didn't in structured natural language, then reads it back. Not fine-tuning, not vector search. Just a curated, ever-improving wiki the agent maintains.

MindMemory implements that loop in the smallest possible form, plus measurement, plus a self-maintenance architecture that runs continuously rather than at session bookends.
